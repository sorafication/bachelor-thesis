Abonnement-Richtlinien für das GraphQL-Schema

Abonnements in AWS AppSync werden als Antwort auf eine Mutation aufgerufen. Das bedeutet, dass Sie jede beliebige Datenquelle in AWS AppSync in eine Echtzeitdatenquelle umwandeln können, indem Sie eine GraphQL-Schemarichtlinie auf einer Mutation angeben. Das AWS AppSync-Client-SDK übernimmt automatisch die Verwaltung der Abonnementverbindung. Das SDK verwendet entweder reine WebSockets oder MQTT über WebSockets als Netzwerkprotokoll zwischen Client und Dienst. Das Protokoll ist von der Version des verwendeten Clients abhängig.

Hinweis: Wenn Sie die Autorisierung zum Zeitpunkt der Verbindung mit einem Abonnement steuern möchten, können Sie Kontrollen wie IAM, Amazon Cognito-Identitätspools oder Amazon Cognito-Benutzerpools für eine Autorisierung auf Feldebene nutzen. Für eine differenzierte Zugriffskontrolle auf Abonnements können Sie Resolver an Ihre Abonnementfelder anhängen und mithilfe der Identität des Aufrufers und der AWS AppSync-Datenquellen eine Logik ausführen. Weitere Informationen finden Sie unter Sicherheit.

Abonnements werden durch Mutationen ausgelöst und die Mutationsauswahlmenge wird an Abonnenten gesendet.

Das folgende Beispiel zeigt, wie Sie mit GraphQL-Abonnements arbeiten. Es wird keine Datenquelle angegeben, da die Datenquelle AWS Lambda, Amazon DynamoDB oder Amazon Elasticsearch Service sein kann.

Für den Einstieg in Abonnements müssen Sie dem Schema folgendermaßen einen Abonnement-Eintrittspunkt hinzufügen: 

schema {
    query: Query
    mutation: Mutation
    subscription: Subscription
}

Angenommen, Sie haben eine Blog-Website und möchten neue Blogs sowie Änderungen an vorhandenen Blogs abonnieren. Fügen Sie zu diesem Zweck folgende Subscription-Definition dem Schema hinzu: 

type Subscription {
    addedPost: Post
    updatedPost: Post
    deletedPost: Post
}

Nehmen wir außerdem an, Sie haben die folgenden Mutationen:

type Mutation {
    addPost(id: ID! author: String! title: String content: String url: String): Post!
    updatePost(id: ID! author: String! title: String content: String url: String ups: Int! downs: Int! expectedVersion: Int!): Post!
    deletePost(id: ID!): Post!
}

Sie können diese Felder in Echtzeitfelder umwandeln, indem Sie folgendermaßen eine @aws_subscribe(mutations: ["mutation_field_1", "mutation_field_2"])-Richtlinie für jedes Abonnement hinzufügen, für das Sie Benachrichtigungen erhalten möchten: 

type Subscription {
    addedPost: Post
    @aws_subscribe(mutations: ["addPost"])
    updatedPost: Post
    @aws_subscribe(mutations: ["updatePost"])
    deletedPost: Post
    @aws_subscribe(mutations: ["deletePost"])
}

Da @aws_subscribe(mutations: ["",..,""]) ein Array von Mutationseingaben erfordert, können Sie mehrere Mutationen angeben, die ein Abonnement auslösen. Wenn Sie das Abonnement von einem Client aus initiieren, sieht Ihre GraphQL-Abfrage möglicherweise wie folgt aus: 

subscription NewPostSub {
    addedPost {
        __typename
        version
        title
        content
        author
        url
    }
}

Die obige Abonnementabfrage wird für Clientverbindungen und Tools benötigt. Wenn Sie jedoch MQTT über WebSockets verwenden, gibt der Client, der die Mutation auslöst, den Auswahlsatz an, den Abonnenten erhalten. Um dies zu demonstrieren, nehmen wir an, dass eine Mutation von einem anderen mobilen Client oder einem Server vorgenommen wurde (z. B mutation addPost(...){id author title }). In diesem Fall werden der Inhalt, die Version und die URL nicht für Abonnenten veröffentlicht. Stattdessen werden die ID, der Autor und der Titel veröffentlicht.

Wenn Sie den reinen WebSockets-Client verwenden, erfolgt die Filterung des Auswahlsatzes pro Client, da jeder Client seinen eigenen Auswahlsatz definieren kann. In diesem Fall muss der Abonnementauswahlsatz eine Teilmenge des Mutationsauswahlsatzes sein. Beispiel: Abonnement addPost(...){id author title url version}, das mit Mutation addedPost{author title} verknüpft ist, erhält nur den Autor und den Titel des Beitrags. Die anderen Felder werden nicht empfangen. Wenn der Mutation jedoch der Autor in ihrem Auswahlsatz fehlte, erhält der Abonnent einen null-Wert für das Autorenfeld (oder einen Fehler, falls das Autorenfeld im Schema als erforderlich/nicht NULL definiert ist).

Wenn Sie MQTT über WebSockets in Ihrer Anwendung verwenden, sind zudem einige Änderungen zu beachten. Wenn Sie den zugeordneten Abonnementauswahlsatz nicht mit den erforderlichen Feldern konfiguriert haben und sich auf die Mutationsfelder verlassen, um Daten an den abonnierten Client zu übertragen, ändert sich das Verhalten, wenn Sie zu reinen WebSockets wechseln. Im obigen Beispiel würde ein Abonnement ohne das in seinem Auswahlsatz definierte Feld „author (Autor)“ den Namen des Autors mit MQTT über WebSockets dennoch zurückgeben, da das Feld in der Mutation definiert ist. Dieses Verhalten gilt nicht für reine WebSockets. Der Auswahlsatz für Abonnements ist bei der Verwendung von reinen WebSockets unerlässlich: Wenn ein Feld nicht explizit im Abonnement definiert ist, wird es von AWS AppSync nicht zurückgegeben.

Im vorherigen Beispiel hatten die Abonnements keine Argumente. Nehmen wir ein Schema wie das folgende an: 

type Subscription {
    updatedPost(id:ID! author:String): Post
    @aws_subscribe(mutations: ["updatePost"])
}

In diesem Fall definiert der Client ein Abonnement wie folgt:

subscription UpdatedPostSub {
    updatedPost(id:"XYZ", author:"ABC") {
        title
        content
    }
}

Der Rückgabetyp eines subscription-Felds in Ihrem Schema muss dem Rückgabetyp des zugehörigen Mutationsfelds entsprechen. Im vorherigen Beispiel war dies als addPost und addedPost dargestellt, zurückgegeben als Post-Typ.

Um Abonnements auf dem Client einzurichten, beziehen Sie sich auf Erstellen einer Client-App.

Verwenden von Abonnement-Argumenten

Bei der Verwendung von GraphQL-Abonnements müssen Sie wissen, wann und wie diese Argumente verwendet werden, weil Sie durch geringfügige Änderungen anpassen können, wie und wann Clients über aufgetretene Mutationen benachrichtigt werden. Halten Sie sich zu diesem Zweck an das Beispielschema im Quickstart-Abschnitt Aufrufen eines Beispielschemas, in dem „Events (Ereignisse)“ und „Comments (Kommentare)“ erstellt werden. In diesem Beispielschema tritt die folgende Mutation auf: 

type Mutation {
    createEvent(
        name: String!,
        when: String!,
        where: String!,
        description: String!
    ): Event
    deleteEvent(id: ID!): Event
    commentOnEvent(eventId: ID!, content: String!, createdAt: String!): Comment
}

Im Standardbeispiel können Clients Kommentare abonnieren, wenn ein bestimmtes eventId-Argument weitergeleitet wird: 

type Subscription {
    subscribeToEventComments(eventId: String!): Comment
        @aws_subscribe(mutations: ["commentOnEvent"])
}

Wenn Sie jedoch zulassen möchten, dass Clients ein einzelnes Ereignis ODER alle Ereignisse abonnieren können, können Sie dies zu einem optionalen Argument machen, indem Sie das Ausrufungszeichen (!) aus dem Abonnement-Prototyp entfernen: 

subscribeToEventComments(eventId: String): Comment

Mit dieser Änderung erhalten Clients, die dieses Argument ausgelassen haben, Kommentare für alle Ereignisse. Wenn Sie möchten, dass Clients explizit alle Kommentare für alle Ereignisse abonnieren, müssen Sie das Argument entfernen: 

subscribeToEventComments: Comment

Nutzen Sie dies für Kommentare zu einzelnen oder mehreren Ereignissen. Wenn Sie über alle erstellten Ereignisse informiert werden möchten, können Sie folgendermaßen vorgehen: 

type Subscription {
    subscribeToNewEvents: Event
        @aws_subscribe(mutations: ["createEvent"])
}

Es können auch mehrere Argumente weitergegeben werden. Wenn Sie beispielsweise über neue Ereignisse an einem bestimmten Ort und zu einer bestimmten Zeit benachrichtigt werden möchten, können Sie folgendermaßen vorgehen: 

type Subscription {
    subscribePlaceDate(where: String! when: String!): Event
            @aws_subscribe(mutations: ["createEvent"])
}

Dies hat zur Folge, dass die Client-Anwendung jetzt Folgendes tun kann:

subscription myplaces {
    subscribePlaceDate(where: "Seattle" when: "Saturday"){
        id
        name
        description
    }
}

Argument-Null-Wert hat Bedeutung

Wenn Sie eine Abonnementabfrage in AWS AppSync erstellen , filtert ein null-Argumentwert die Ergebnisse anders, anstatt das Argument vollständig auszulassen.

Wir erläutern dies nun anhand eines Beispiels. Gehen wir zurück zum Beispiel für die Ereignisse-App, wo wir Ereignisse erstellen und Kommentare zu Ereignissen veröffentlichen können. Weitere Informationen finden Sie im Beispielschema im Quickstart-Abschnitt Aufrufen eines Beispielschemas.

Lassen Sie uns unser Schema ändern, um ein neues location-Feld in das Comment-Feld aufzunehmen, das beschreibt, woher der Kommentar gesendet wurde. Der Wert kann ein Satz von Koordinaten oder ein Ort sein. Beachten Sie, dass wir das nachstehende Schema der Kürze halber vereinfacht haben: 

type Comment {
    # The id of the comment's parent event.
    eventId: ID!
    # A unique identifier for the comment.
    commentId: String!
    # The comment's content.
    content: String
    # Location where the comment was made
    location: String
}

type Event {
    id: ID!
    name: String
    where: String
    when: String
    description: String
}

type Mutation {
    commentOnEvent(eventId: ID!, location: String, content: String): Comment
}

type Subscription {
    subscribeToEventComments(eventId: String!, location: String, content: String): Comment
        @aws_subscribe(mutations: ["commentOnEvent"])
}

Beachten Sie das neue optionale Feld Comment.location.

Nehmen wir nun an, dass wir über alle Kommentare benachrichtigt werden möchten, wenn sie für ein bestimmtes Ereignis veröffentlicht werden, würden wir das folgende Abonnement schreiben: 

subscribeToEventComments(eventId: "1") {
    eventId
    commentId
    location
    content
}

Wenn wir nun stattdessen das Feldargument location: null zum Abonnement oben hinzufügen würden, 

subscribeToEventComments(eventId: "1" location: null) {
    eventId
    commentId
    location
    content
}

würden wir nun eine andere Frage stellen. Dieses Abonnement registriert den Client jetzt, um über alle Kommentare benachrichtigt zu werden, die keinen Speicherort für ein bestimmtes Ereignis angegeben haben.
